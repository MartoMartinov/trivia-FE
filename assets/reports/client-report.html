01

Difficulty Multiplier Progression

"I feel like even when getting the right question, I was shown easy questions in a row, and after a few then immediately hard."

Working as Designed

How it works

Difficulty is streak-driven, not count-driven. The next question's difficulty is calculated in game.updaters.ts → selectNextDifficulty() after each answer:

Outcome Streak Next Difficulty Multiplier
Wrong resets to 0 easy 1×
Correct 1 easy 1×
Correct 2–3 medium 1.5×
Correct 4–5 hard 2×
Correct 6+ hard_plus 2.5×

Why it looked wrong

The player gets two easy questions before the first jump (streak 1 is still easy). After streak 2, it jumps to medium, after streak 4 to hard. Any wrong answer resets to easy — so a player who got easy, easy, medium, then missed, would restart at easy and could hit hard quickly once they recovered their streak.

Client response

The behavior is working as designed. The progression is streak-based: you need to answer 2 in a row correctly to reach medium, 4 in a row for hard. A wrong answer resets the streak to easy immediately. If the client would prefer a different shape — e.g. a fixed sequence (Q1 easy → Q2 medium → Q3 hard) regardless of performance — we can adjust the thresholds, but that would change the scoring mechanics too. Let us know if they want a change and we'll scope it.
02

Question Load Time / Abrupt Session Stop

"Sometimes the session stops abruptly before the timer reaches 0, and the next question takes too long to load."

Bug Confirmed

Buffer architecture

Session start: BE returns currentQuestion (shown immediately) + buffer[] (one question per difficulty — ready for Q2).

After each answer: Q(n+1) is picked from the buffer instantly, then the buffer is cleared and fetchNextBatch() fires in the background to pre-load Q(n+2).

Race condition (root cause)

When the player answers Q2 before fetchNextBatch has returned, the buffer is empty. pickFromBuffer([]) returns null. The store sets currentQuestion = null, hasMoreQuestions becomes false, and the game navigates to sponsor/results — even though questions remain.

This is the abrupt ending. It happens reliably on slow connections or fast players.

Fix required

The store should not treat an empty buffer as "no more questions." Options: (a) hold a loading state when the buffer is empty and currentIndex < totalQuestions, showing a brief spinner instead of ending the game; (b) keep the picked question in the buffer until the next batch arrives before clearing. Will add this to Asana.

Client response

Confirmed — we found the cause. Under slow network conditions the game can incorrectly treat a temporary empty question buffer as "session over" and skip to the results screen. This is a bug, not expected behavior. We have identified the fix and will deploy it in the next release.
03

Sponsor Hit — Dashboard Metric

"What does this mean? How many times was the sponsor question answered correctly?"

Clarification

What it tracks — clarified by BE dev

Sponsor hit % = the percentage of players who clicked the sponsor's website link during the sponsor round. It maps to the website_click event tracked in the backend, not to correct answers on the sponsor question (those are tracked separately).

Why it shows 0%

No player has tapped the sponsor website link yet — either because no full sessions with a sponsor round have been completed, or players reached the sponsor screen but did not click through to the site. Not a bug.

Client response

"Sponsor hit %" tracks how many players clicked the sponsor's website link during the bonus round — it is a click-through rate, not a measure of correct answers. 0% means no player has tapped that link yet, most likely because no full sessions including the sponsor round have been completed during testing. Once real game sessions run at the event, this metric will update automatically.
04

Geo-targeting — IMTS / McCormick Place Radius

"How hard would it be to geo-target the trivia for IMTS only to McCormick Place/Chicago radius?"

Low Priority — Assess

Current access control

The /register route has no guard — anyone with the URL can register and play. The game routes (/game, /sponsor, /results) require authentication, but registration itself is open. There is no hash or token in the URL for the player flow.

Options

Approach Effectiveness Complexity Notes
On-site PIN / access code at registrationRecommended Good Low 4-digit code printed on badges / stands. Simple BE validation field at registration.
QR code rotation (change URL/hash periodically) Partial Low Shared links expire, but doesn't truly confirm physical presence.
GPS geofencing (Geolocation API, ~500 m radius) High Medium Works on mobile. Users who deny location permission would be blocked. Indoor GPS can drift.
IP-based geofencing (venue WiFi ranges) Low High Requires obtaining McCormick's IP ranges. VPN / cellular easily bypasses.

Client response

The easiest approach without heavy infrastructure is an on-site PIN code required at registration — a short code printed on badges, stands, or handed out at the booth that proves physical attendance. GPS geofencing (~500 m radius around McCormick Place) is also feasible on the frontend and would work for most mobile users, though anyone who denies location permission would be blocked. Since the client flagged this as low priority, we recommend the PIN approach if they want to proceed — it is a 1–2 day implementation. We can scope GPS geofencing separately if they want a stronger guarantee.
